> 并非勾史

有一篇论文, <<Building a Secure Software Supply Chain with GNU Guix>>, 讲供应链的每个环节, 从源代码到二进制都有被攻击的可能

传统包管理有着二进制分发, 依赖中央仓库, 难以验证构建过程是否真实的局限性
缺乏对 Downgrade Attacks(回滚式攻击) 的防御, 缺乏对 Code Provenance(代码溯源) 的认证

什么叫 Downgrade Attack? 是一种利用系统/协议(网络协议中也有喵)的向下兼容性, 使得攻击者完成了通过降低安全级别进行攻击的手段
举个例子, 本地安装的 openssl-3.0 此时已修复漏洞 CVE-2023-1234, 但攻击者篡改了仓库的 metadata 且伪造了数字签名
此时, 看似是 openssl-3.1, 其实只是被伪造的, 还未修复漏洞时的 openssl-2.9, 更新变成了迈向较低安全性的一次降级

在 guix-manual 的第 6.6 节(Invoking guix pull)中的 --allow-downgrades 下面, 提及过 guix 在默认情况下可以阻止 Downgrade Attacks
除非你在明确风险后运行 guix pull --allow-downgrades, 才会使得 channel 的 git 仓库被重置为较早版本, 导致安装较旧且已知易受攻击的包

传统的包管理工具并没有针对这种情况的版本连续性检查机制, 更新时仅验证版本号而非时间顺序

🔗 系统安装

> 下载ISO

首先, 让我们从官网的 地址 上, 把 ISO 文件下载到本地
你也可以通过 https://mirrors.sjtug.sjtu.edu.cn/gnu/guix/guix-system-install-X.Y.Z.x86_64-linux.iso 在上交的镜像站上下载
比如在我写下这篇文章的 2025-06-16 时, Guix System 的最新发行版本是 1.4.0, 那么可以:

wget https://mirrors.sjtug.sjtu.edu.cn/gnu/guix/guix-system-install-1.4.0.x86_64-linux.iso
curl -O https://mirrors.sjtug.sjtu.edu.cn/gnu/guix/guix-system-install-1.4.0.x86_64-linux.iso
axel https://mirrors.sjtug.sjtu.edu.cn/gnu/guix/guix-system-install-1.4.0.x86_64-linux.iso

然后我们就能用 qemu/kvm, 还有各种其他虚拟机来体验 guix, 当然也可头铁地直接在实体机上安装
但是极其不推荐! 强烈建议先在虚拟机上使用, 熟悉基本操作后再考虑实体安装, 而且务必记得去看下 nonguix 的文档
当我们在虚拟机上熟悉后, 可以尝试在服务器上使用, 毕竟 guix/nix 声明式的配置, 其稳定程度坚若磐石

我将虚拟机的启动方式配置为 UEFI, 现在启动后会出现 grub 菜单, 选中第一个 GNU Guix installation 1.4.0, 回车进入
Locale language/location 随便选一个, 然后选择 Install using the shell based process 来手动安装
(因为我们需要设置镜像源与添加更多自用配置, 所以不使用 TUI 工具, 而是直接在 shell 里面手动敲命令装)

> SSH连接

TTY 中的黑色框框好丑呀! 先用 ssh 登录虚拟机
Guix 中类似 systemd/openrc/runit 的初始化系统/服务管理器叫 shepherd(牧羊人)
这是用一种 scheme 方言(gnu/guile) 写的, 听上去就觉得好酷欸不是吗 (x

让我们使用 herd 开启 sshd 服务:

herd status sshd
herd start sshd

在虚拟机中为 root 设置 password: passwd root
在宿主机上用 ssh 连接上在虚拟机中用 ip a 得到的 IP 地址:

ssh root@192.168.122.14

(可选) 用 tmux 来避免 ssh 意外断开而导致任务退出(推荐这样做, 尤其是在 guix pull 的时候, 因为好慢!):

# 用镜像源安装 tmux
GUIX_BUILD_OPTIONS="--substitute-urls=https://mirror.sjtu.edu.cn/guix" guix install tmux

# 或者这样用镜像源安装 tmux
guix install tmux --substitute-urls=https://mirror.sjtu.edu.cn/guix

# ssh 成功后用 tmux 开启一个绘会话
tmux new-session -s guix-session

# ssh 意外断开, 则用 ssh 再次登录后, 通过 tmux 重连已有会话
tmux attach-session -t guix-session

> 磁盘分区

我使用 cfdisk 进行分区, 是 UEFI + GPT 的组合, 常见于新电脑:
(以防有小白不了解, 我们启动模式选的是 UEFI, 所以分区表那里选择 gpt)

cfdisk /dev/vda

然后分别对其进行初始化:

最后进行挂载工作:

# 挂载根分区
mount -L my-root /mnt

# 挂载 ESP 分区 (UEFI)
mkdir -p /mnt/boot/efi
mount /dev/vda1 /mnt/boot/efi

(可选) 使用 swap 分区:

mkswap /dev/vda3
swapon /dev/vda3

(可选) 倘若你不想专门设置 swap 分区, 也可使用 swap 文件:

# This is 10 GiB of swap space.  Adjust "count" to change the size.
dd if=/dev/zero of=/mnt/swapfile bs=1MiB count=10240

# For security, make the file readable and writable only by root.
chmod 600 /mnt/swapfile
mkswap /mnt/swapfile
swapon /mnt/swapfile

> 继续安装

此时, 我们的根分区已经挂载到 /mnt, 随后执行:

herd start cow-store /mnt

这将设置 /gnu/store 为写入时复制(copy-on-write), 以便安装时被添加的软件包被写入 /mnt, 而非保存在内存中而导致内存不足
因为下文中的 guix system init 命令, 在第一阶段需向 /gnu/store(最初是一个内存文件系统) 下载/构建内容

接下来我们需要编辑文件, 以提供待安装系统的声明式配置
官方的 ISO 中可以使用 nano, mg(emacs clone), nvi(vi clone) 命令进行编辑, 我们下文将使用 nano:

mkdir /mnt/etc
nano /mnt/etc/config.scm

务必记得 将配置文件存储在目标根目录下, 比如 /mnt/etc/config.scm, 不然当重启到新系统后, 它就丢失了

以下是一份作为示例的 /mnt/etc/config.scm:

;; This is an operating system configuration template
;; for a "bare bones" setup, with no X11 display server.

(use-modules (gnu) (guix channels))
(use-service-modules networking ssh)
(use-package-modules screen ssh)

;; 不推荐启用如下配置, 请配置下一节里面说的用户级别的 channels
;; 我们可以在系统配置文件中配置 `/etc/guix/chanels.nix`
;; 目前镜像站中只有 nju 的 channel 可访问的, 但也很不稳定 (我 tm 下到一半报错退出了啊啊啊!!!)
;; 所以这里直接用后续会成为默认 channel 的 codeberg 了
;; (define my-channels (list (channel
;;   (inherit (car %default-channels))
;;   (url "https://codeberg.org/guix/guix.git"))))

(operating-system
  (host-name "guix")
  (timezone "Asia/Shanghai")
  (locale "en_US.utf8")

  ;; Boot in "legacy" BIOS mode, assuming /dev/sdX is the
  ;; target hard disk, and "my-root" is the label of the target
  ;; root file system.
  (bootloader (bootloader-configuration
    (bootloader grub-efi-bootloader)
    (targets '("/boot/efi"))))
  ;; It's fitting to support the equally bare bones ‘-nographic’
  ;; QEMU option, which also nicely sidesteps forcing QWERTY.
  (kernel-arguments (list "console=ttyS0,115200"))
  (file-systems (cons
    (file-system
      (device (file-system-label "my-root"))
      (mount-point "/")
      (type "btrfs"))
    %base-file-systems))

  ;; This is where user accounts are specified.  The "root"
  ;; account is implicit, and is initially created with the
  ;; empty password.
  (users (cons
    (user-account
      (name "kawayww")
      (comment "I am KawaYww!")
      (group "users")
      ;; Adding the account to the "wheel" group
      ;; makes it a sudoer.  Adding it to "audio"
      ;; and "video" allows the user to play sound
      ;; and access the webcam.
      (supplementary-groups '("wheel" "audio" "video" "wheel" "input")))
     %base-user-accounts))

  ;; Globally-installed packages.
  (packages (cons screen %base-packages))

  ;; Add services to the baseline: a DHCP client and
  ;; an SSH server.
  (services (cons*
    (service dhcp-client-service-type)
    (service openssh-service-type
      (openssh-configuration
      (openssh openssh-sans-x)
      (port-number 2222)))
    ;; 这里注释掉了下面默认的 `%base-services`
    ;; 其中的 channel 与 substitute-urls 被替换为了镜像地址
    ;; %base-services)))
    (modify-services %base-services
      (guix-service-type config =>
        (guix-configuration
          (inherit config)
          (http-proxy "http://192.168.122.1:7890")
          ;; 如该配置文件上头所说, 不推荐在这里更改 channels
          ;; (channels my-channels)
          ;; (guix (guix-for-channels my-channels))
          (substitute-urls (cons*
            "https://mirror.sjtu.edu.cn/guix/"
            "https://mirror.nju.edu.cn/guix/"
            "https://mirrors.tuna.tsinghua.edu.cn/guix/"
            %default-substitute-urls)))))))
)

上面的声明式配置, 其大多地方不言而喻, 而其中 channels 与 substitute-urls 的简单概念如下:

  • channels: 提供包/服务的定义, 包含软件包的描述, 版本等
  • substitute-urls: 提供预编译的产物, 避免用户每次都要自己重新编译


当编辑好配置文件后, 记得初始化系统(目标根目录挂载于 /mnt 下):

guix system init /mnt/etc/config.scm /mnt --substitute-urls="https://mirror.sjtu.edu.cn/guix/"

然后记得 passwd <your-user-name>, 来设置用户的登录密码
上面声明的 port-number 是 2222, 所以装好后如果想 ssh, 应该用端口 2222:

ssh -p 2222 kawayww@192.168.122.14

初始安装完毕后, 当你再次修改了 /etc/config.scm 并想让配置生效, 请输入如下命令

sudo -E guix system reconfigure /etc/config.scm

注意:
sudo guix 运行的是当前用户的 guix 命令, 而非 root 下的 guix (sudo 保持环境变量 PATH 不变)
想显式运行 root 下的 guix, 需要使用 sudo -i

注意:
guix pull 仅针对运行它的用户更新 guix 命令与包定义
若你想在 root 下用 guix system reconfigure, 则需单独进行 guix pull

注意:
guix pull 只吃环境变量 http(s)_proxy, 不吃配置文件里 guix-configuration 下声明式的 http-proxy)
此时需要 sudo -E 使用 user 定义的环境变量, 如 http_proxy/https_proxy

> 镜像源加速

提示: 这一步有点蛋疼, 主要在于 guix pull 这块

可参考官方文档(还有好多网络问题/坑点是自己摸出来的呜呜呜): Getting-Substitutes-from-Other-Servers
可用的镜像源地址如下(写下本文时, 只有 nju 的 channel 与 substituter 是都有效的):

  • https://mirror.nju.edu.cn/git/guix.git
  • https://mirror.sjtu.edu.cn/git/guix.git
  • https://mirrors.tuna.tsinghua.edu.cn/git/guix.git


目前镜像站的 substitute 比较完备, 但对 channels 的支持很崩, 建议直接用官方迁移至 codeberg.org 上的仓库(后续将成为默认):
我们可以直接用环境变量来提供 substitute(用国内源安装预编译产物来加快构建), http(s)_proxy(为 guix pull 准备代理):

export GUIX_BUILD_OPTIONS="--substitute-urls=https://mirror.sjtu.edu.cn/guix"
export http_proxy="http://192.168.122.1:7890"
export https_proxy="http://192.168.122.1:7890"

如果我们想更新 channel, 那么就需要 guix pull
但 guix pull 不吃 guix-daemon 设置的 http-proxy 设置(但其他命令比如安装包时会用)
所以对于 guix pull, 我们仍需单独设置环境变量: http_proxy/https_proxy

而除了 guix pull 的其他 guix 系列的命令, 都认配置文件里声明式的 http-proxy
(虚拟机连本地代理记得让本机防火墙 allow 一下代理端口, 比如 7890)

sudo herd set-http-proxy guix-daemon http://192.168.122.1:7890

我们可以把 guix pull 这样写成一行, 但是不推荐:

http_proxy=http://192.168.122.1:7890 http_proxy=http://192.168.122.1:7890 guix pull --url=https://codeberg.org/guix/guix --substitute-urls=https://mirror.sjtu.edu.cn/guix

推荐的方式是修改 ~/.config/guix/channels.scm:

(list (channel
  (inherit (car %default-channels))
  (url "https://codeberg.org/guix/guix.git")))

如果直接用默认的 channel(目前并非 codeberg, 而是勾史的 https://git.savannah.gnu.org), 则会遇见 Git error: SSL error: xxxxx
等 codeberg 成为默认的了, 只需要添加 http(s)_proxy 环境变量后, 即可直接 guix pull 了

在 ~/.config/guix/channels.scm 里配置的 channels 会在 guix pull 时被自动读取
在写入该文件后, 每次更新 channels 与系统配置的标准流程如下(一般并不需频繁调用 guix pull 来更新包定义):

guix pull
sudo -E guix system reconfigure /etc/config.scm

如果在 guix pull 期间遇见类似下面这些报错:

guix pull: error: directory '/home/kawayww/.cache/guix' is not owned by user kawayww
hint: You should run this command as root; use `sudo -i' or equivalent if you really want to
pull as kawayww.
Updating channel 'guix' from Git repository at 'https://codeberg.org/guix/guix.git'...
guix pull: error: mkdir: Permission denied

那么你可以运行如下命令解决权限问题:

sudo chown -R $USER ~/.cache/guix

由于需要代理来访问 codeberg.org, sudo -E 导入用户定义的 http_proxy/https_proxy 环境变量是必要的
不建议把 channels 的配置写在 /etc/config.scm 里, 不然无法享用代理

对于运行单个命令时, 也可以临时覆盖 substitube 服务器:
(字符串那里你可以添加多个 substitute-url, 用空格进行分割)

guix package -i <package> --substitute-urls="https://mirror.sjtu.edu.cn/guix/"

对于使用 Guix 系统的, 请编辑 /etc/config.scm 文件
将 services 中最后的 %base-services 替换为下面的 (modify-services $base-services (...)):

  (services (cons*
    (service dhcp-client-service-type)
    (service openssh-service-type
      (openssh-configuration
      (openssh openssh-sans-x)
      (port-number 2222)))
    ;; 这里注释掉了下面默认的 `%base-services`
    ;; 其中的 channel 与 substitute-urls 被替换为了镜像地址
    ;; %base-services)))
    (modify-services %base-services
      (guix-service-type config =>
        (guix-configuration
          (inherit config)
          (http-proxy "http://192.168.122.1:7890") ;; 换成你自己的代理服务器
          ;; 不推荐在这里更改 channels
          ;; (channels my-channels)
          ;; (guix (guix-for-channels my-channels))
          (substitute-urls (cons*
            "https://mirror.sjtu.edu.cn/guix/"
            "https://mirror.nju.edu.cn/guix/"
            "https://mirrors.tuna.tsinghua.edu.cn/guix/"
            %default-substitute-urls)))))))
